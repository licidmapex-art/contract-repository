Intelligent Contract Repository — Pilot Build Spec

0. Goal of the pilot

Prove the core workflow end-to-end, on free-tier infrastructure only, for a small real (or realistic) set of contracts (10–50 documents):

Upload (UI or email) → AI extracts metadata → human confirms low-confidence fields only → tidy filename applied → contract searchable/filterable in a dashboard → contract readable in-app with search → basic Q&A over the data.

Explicitly out of scope for the pilot (phase 2+): Teams bot, semantic (vector) search over document text, multi-field playbook editor UI, relationship graph UI polish, SharePoint/Azure SQL variants.

2. Data model (Postgres / Supabase)

sql
-- Contracts are "cases" — a container, not a single file
create table contracts (
  id uuid primary key default gen_random_uuid(),
  title text,                      -- optional human label, can be AI-suggested
  status text not null default 'active' check (status in ('active','inactive')),
  -- raw stored status only; "expiring"/"expired" are COMPUTED, not stored
  created_at timestamptz default now()
);

-- Every uploaded file belongs to a contract, tagged by its role in the bundle
create table documents (
  id uuid primary key default gen_random_uuid(),
  contract_id uuid references contracts(id) on delete cascade,
  role text not null check (role in ('original','annex','amendment','renewal','correspondence','other')),
  storage_path text not null,      -- path in Supabase Storage
  original_filename text not null, -- never overwritten — audit trail
  generated_filename text,         -- tidy name, once naming convention applied
  extracted_text text,             -- full text pulled from the PDF, used for search/Q&A
  uploaded_via text default 'ui',  -- 'ui' | 'email'
  created_at timestamptz default now()
);

-- The "playbook" catalog: built-in fields and user-defined custom fields
-- live in ONE system, no structural difference between them
create table metadata_fields (
  id uuid primary key default gen_random_uuid(),
  key text unique not null,            -- e.g. 'counterparty', 'regulated'
  label text not null,                 -- display name
  category text not null,              -- 'core' | 'financial' | 'compliance' | 'custom'
  field_type text not null check (field_type in ('text','date','number','boolean','enum')),
  enum_options text[],                 -- only used when field_type = 'enum'
  playbook_prompt text not null,       -- instructions given to the AI for this field
  is_builtin boolean default false,
  created_at timestamptz default now()
);

-- The actual extracted (or manually entered) value, per contract, per field
create table metadata_values (
  id uuid primary key default gen_random_uuid(),
  contract_id uuid references contracts(id) on delete cascade,
  field_id uuid references metadata_fields(id) on delete cascade,
  value text,                          -- store as text, cast in application layer by field_type
  confidence numeric,                  -- 0–1, null if manually entered
  confirmed boolean default false,
  source_document_id uuid references documents(id), -- where the AI found this, for citation
  updated_at timestamptz default now(),
  unique (contract_id, field_id)
);

-- Typed links between contracts — a small graph, not a field
create table contract_relationships (
  id uuid primary key default gen_random_uuid(),
  contract_a_id uuid references contracts(id) on delete cascade,
  contract_b_id uuid references contracts(id) on delete cascade,
  relationship_type text not null,     -- 'renews' | 'supersedes' | 'same_counterparty' | 'related'
  confirmed boolean default false,
  created_at timestamptz default now()
);

-- Naming convention template (single row for pilot; org-level setting later)
create table naming_settings (
  id uuid primary key default gen_random_uuid(),
  template text not null default '{contract_type}_{counterparty}_{effective_date}_{document_role}',
  keep_original_name boolean default true
);

Seed metadata_fields with built-ins so the system has content on day one: contract_type, counterparty, effective_date, expiry_date, value, notice_period_days — all category = 'core', is_builtin = true.

Derived status logic (compute in application code, not stored):

effective_status(contract):
  if contract.status == 'inactive': return 'inactive'
  if expiry_date is null: return 'active'
  if expiry_date < today: return 'expired'
  if expiry_date <= today + 60 days: return 'expiring'
  return 'active'

3. Extraction pipeline

  1. File uploaded (UI or email) → stored in Supabase Storage → documents row created.
  2. Extract raw text (native PDF → direct text extraction; scanned → OCR step, e.g. pdf-parse + fallback OCR library — flag as a pilot stretch goal if scanned docs are rare in your test set).
  3. For each metadata_field, call Gemini with a prompt built as:
   You are extracting a specific field from a contract.
   Field: {label}
   Instructions: {playbook_prompt}
   Return ONLY valid JSON: {"value": "...", "confidence": 0.0-1.0}
   If the field cannot be determined, return {"value": null, "confidence": 0}.

   Document text:
   """{extracted_text}"""
  1. Batch fields into as few calls as practical (send all core fields in one call with a combined JSON schema, to save on requests) — write one metadata_values row per field either way.
  2. If confidence >= 0.75 → confirmed = true automatically. If confidence < 0.75 → confirmed = false, surfaced in a review queue.
  3. Once all fields for a contract are either confirmed or auto-confirmed, apply the naming convention (§5) to generate generated_filename.

4. API routes (Next.js API routes on Netlify)

  • POST /api/contracts/upload — accepts file(s) + contract_id (or creates new contract), triggers extraction pipeline
  • GET /api/contracts — list with query params for filters (?status=active,expiring&field:regulated=true)
  • GET /api/contracts/:id — full detail incl. documents, metadata, relationships
  • POST /api/contracts/:id/metadata/:fieldId/confirm — human confirms/edits a value
  • GET /api/metadata-fields — list all field definitions, grouped by category (powers the onion filter's first layer)
  • POST /api/metadata-fields — create a new custom field + playbook prompt; triggers re-run of extraction for that field across existing contracts (pilot: can be manual "Run now" button rather than automatic)
  • POST /api/relationships — link two contracts with a type
  • POST /api/ingest/email — webhook or polling endpoint that pulls attachments from the contracts inbox (see §6) and forwards them to the upload pipeline
  • POST /api/ask — pilot Q&A: translates a natural-language question into a filter query against metadata_values (see §7 — no vector search yet)

5. Naming convention

Apply naming_settings.template, substituting tokens from confirmed metadata_values plus document.role. Collision handling for the pilot: if two documents would generate the same filename, append _2, _3, etc. Always keep original_filename untouched.

6. Email ingestion (pilot-simple version)

  • Use a free Gmail address (or your existing domain's inbox) dedicated to contracts.
  • Poll it periodically (e.g. every few minutes via a scheduled Netlify function, or a simple cron-triggered script) using the Gmail API — free tier is generous enough for pilot-scale polling.
  • For each new email: extract PDF attachments → call /api/contracts/upload for each → optionally send an auto-reply confirming receipt and the generated filename once processing completes.

7. Pilot-scope Q&A (structured only, no vector search yet)

For the pilot, skip semantic search over document text (needs an embeddings store, extra complexity) and support structured questions only:

  1. Send the question + the list of available metadata_fields (key, label, type) to the AI, asking it to return a structured filter:
json
   {"filters": [{"field": "expiry_date", "op": "<=", "value": "2026-09-30"}]}
  1. Run that filter against metadata_values / contracts.
  2. Feed the matching contracts back to the AI to produce a natural-language answer, citing contract titles.

This covers "which contracts expire this quarter" style questions well and is a fraction of the complexity of full semantic search — a good phase-2 upgrade once the pilot proves the core loop.

8. UI build order (for Cursor)

  1. Contract list/dashboard (table + status chips + metric cards)
  2. Upload flow (drag-drop → progress → extraction status)
  3. Contract detail page (documents bundle, metadata panel, relationships)
  4. Onion filter panel (category → field → value, active-filter chips)
  5. In-app reader with search bar (react-pdf + text layer)
  6. Review queue (low-confidence fields awaiting confirmation)
  7. Naming convention settings page
  8. Simple Q&A input box wired to /api/ask

9. Known pilot-stage limitations to flag to stakeholders

  • No automated backups (Supabase free tier) — acceptable for a pilot, not for production data.
  • Supabase project pauses after 7 days of no activity — a demo scheduling gotcha, not a bug.
  • Gemini free tier may be geographically restricted (EU) — confirm before relying on it; budget a few dollars for paid tier if needed.
  • No semantic/content-level search yet — only structured metadata questions.
  • Scanned/image-only PDFs need an OCR step not detailed above — treat as a stretch goal if your pilot set is mostly native-text PDFs.
